# ai-workflow.md

How this prototype was built with AI tooling, and how I kept it honest.

## Which AI tools and models did I use, and for what?

- **Claude Code (Anthropic, Claude Opus)** as the primary build agent — reading
  the brief and `Dataset Guide.md`, exploring the CSVs, designing the
  deterministic-first architecture, and writing `data.py`, `metrics.py`,
  `tools.py`, `profile.py`, `assistant.py`, the Streamlit/CLI front-ends, and
  the test suite. I drove it: I made the architectural calls (deterministic
  numbers vs LLM language, investor-scoping at a single choke point, how to
  treat pending capital) and used the agent to implement and verify them fast.
- **Claude (`claude-sonnet-5`, `claude-opus-4-8` configurable)** is the model
  the prototype itself uses at runtime — for tool routing and writing the
  personalised answer. Chosen for reliable tool-calling and good
  instruction-following on the "never invent numbers, cite sources, adapt tone"
  constraints. Sonnet is the cost/latency default; Opus is a one-line swap.

Before writing any metric, I had the agent probe the data directly (latest
valuation per deal, the Forgecraft multi-round case, Grace Okafor's pending
allocation, the Helianthe exit, the Tallybook secondary) so the formulas were
grounded in the real rows, not assumptions.

## Roughly what percentage of the code was AI-generated?

**~90%.** Almost all of the Python and docs were generated by the agent. My
contribution was the design and the judgment calls: the deterministic/LLM split,
the source-citation contract (`{data, sources}` from every tool), the
investor-scoping guarantee, the pending-capital treatment, the tool taxonomy,
and the choice of edge cases to test. I reviewed and corrected the output as it
went rather than accepting it blind.

## What did I reject or materially change from AI suggestions, and why?

- **Pending/unfunded allocation as "value".** The first computation counted
  Grace Okafor's unfunded Helixar Bio units in current value and produced a
  divide-by-zero-ish MOIC. I rejected that — unfunded capital isn't deployed —
  and changed it to exclude pending allocations from value/MOIC and surface them
  separately as a pending commitment. (Now a regression test.)
- **Letting the model compute or carry numbers.** I kept *all* arithmetic in
  Python and made the system prompt forbid the model from calculating or reusing
  figures across turns. The model only routes and writes.
- **"Just build the web app."** I insisted on an **offline deterministic
  fallback** so the prototype runs and the data layer is demonstrable even with
  no API key — the brief explicitly penalises "impossible to run."
- **Current-value rule for exits/write-offs.** Generic `units × latest price`
  would mis-mark a written-off or exited position; I pinned it to 0 for
  Written-Off and netted out the realised fraction so partial secondaries
  (Tallybook) keep their live remainder.
- **Citations scope.** Statements can cite hundreds of rows; I capped the
  `sources` payload to keep answers readable and tokens bounded.

## How did I verify the assistant's answers were correct?

1. **A 17-test verification suite** (`./run.sh test`, runs offline) asserting the
   *named* Dataset-Guide edge cases produce correct, investor-honest numbers:
   multi-round aggregation, share/fee discounts, multi-currency FX symmetry,
   pending, zero-holding, exit (carry + MOIC), write-off (value 0 + loss), down
   round, partial secondary, fee-discount-flag consistency, obligations vs the
   report date.
2. **Internal-consistency checks** independent of any expected constant:
   aggregate position value == sum of its rounds; statement capital contributions
   == sum of paid capital calls; A→B→A FX round-trips exactly.
3. **Manual cross-checks** during the build — e.g. hand-computing INV001's
   portfolio (£176,296 committed / £168,593 contributed / MOIC 2.601x) and
   confirming the running prototype matched.
4. **Runtime grounding** — every numeric answer ends with a `Source:` line, so a
   reviewer can trace any figure to a CSV row; the model is structurally unable
   to reach another investor's data.

## If I had an autonomous coding agent for another 8 hours, where would I point it?

1. **An answer-faithfulness checker (highest value):** post-process every LLM
   reply to confirm each number in the prose appears in a tool result; flag/redo
   on mismatch. This closes the one remaining gap — the model mis-stating a
   correct figure.
2. **An LLM-as-judge eval harness:** a labelled question bank across all
   investors and edge cases, scoring grounding, correctness, refusal of
   cross-investor queries, and tone-appropriateness; run it in CI.
3. **Better entity resolution:** embeddings/fuzzy matching over company, deal and
   sector names (handling the Northpeak Analytics vs Health trap robustly) with a
   disambiguation prompt when ambiguous.
4. **Conversational memory + proactivity:** carry context across turns safely,
   and surface unprompted nudges (overdue fees, upcoming calls) — the first step
   toward the relationship-manager bot in `ROADMAP.md`.
5. **Streaming + richer UI:** token streaming, inline source highlighting, and a
   one-click "show me the rows" drill-down.
